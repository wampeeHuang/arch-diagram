# arch-diagram 复盘：为什么程序化架构图生成失败了

2026-06-18

## 目标

做一个 JSON 驱动的架构图生成器——用户写定义 JSON，渲染引擎产出整洁的 Phase 分组 + 节点 + 边线的流程图。核心使用场景是可视化 Claude Code 技能的流水线。

## 做了什么

- 两阶段布局：Dagre 做 Phase 级宏观布局 → CSS 绝对定位做 Phase 内节点排列
- 真实 DOM 测量（隐藏 off-screen 渲染 → offsetWidth/Height），不用字符估算
- 边线路由四模式：同行水平、同 Phase 跨行、跨 Phase L 形、回退/循环右侧绕行
- 色板对齐 diagram-design 的 style-guide，CSS 自定义属性统一管理
- 三套模板：skill-pipeline (6 节点)、decision-branch (7 节点)、feedback-loop (4 节点)

## 验证结果

程序化指标全部通过：
- 无节点重叠、无边线穿节点
- 无控制台报错
- Phase 间距均匀 (64px)、节点间距均匀 (16px)、路径坐标干净

## 为什么视觉不通过

**1. 尺度体系不成立**

当前参数是一组离散的像素值：线宽 1.5px、箭头 6×4、字体 12px、节点 padding 14px、Phase 间距 64px。它们之间没有比例关系。好的排版建立在"基准单位 × 倍数"上（如 4px grid × 1/2/3/4 倍），而不是各自独立的绝对数值。没有比例系统，视觉必然"说不清哪里不对但就是不对劲"。

**2. 留白无法量化**

程序能算出"不重叠"，但算不出"舒服"。16px 的节点间距不过叠，但看起来拥挤还是松弛，取决于节点大小、字体、边框粗细的共同作用。这本质是一个设计判断，不是算法问题——同样的间距在一种节点密度下刚好，在另一种下就是挤。

**3. 布局引擎没有审美**

Dagre 做的是"找到一个合法的节点排列"，目标是边线最少交叉。但设计意义上的好构图要的是：节奏感（Phase 大小对比）、呼吸感（留白分布）、视觉流向（读者的眼动路径）。Dagre 只知道 x/y 坐标，不知道这些。

**4. 边线标签是事后缝合**

当前标签位置取路径中点的 ±10px 偏移。这带来了两个问题：一是标签可能落在节点边缘上（decision-branch 中"深度""简版"距 decide 底边仅 2px），二是标签与路径的相对位置不统一——有时在路径上方，有时在线段转折处被夹住。真正的边线标签应该是布局的一等公民，在计算节点位置时就预留空间，不是画完路径再贴上去。

**5. 菱形/特殊形状的锚点不精确**

当前用矩形包围盒的 center 和 edge 作为锚点，对 rect 有效，对 diamond 只能算近似。diamond 的实际可见边界比包围盒小，导致边线在视觉上"没碰到"节点的感觉。精确锚点需要做 polygon edge intersection。

## 核心结论

**这个任务本质上是一个排版引擎，不是一个布局算法。**

浏览器已经有最好的排版引擎（CSS Box Model + Flexbox + Grid），但它不暴露"所有 box 精确位置算完后的坐标"给 JS。等到 `getBoundingClientRect()` 能读坐标时，浏览器已经渲染完了——属于"读结果"而非"参与排版"。

反过来，Canvas/SVG 模式可以精确控制每个像素，但要自己实现文字测量、换行、盒模型——等于从零写一个排版引擎。

**两难困局**：用 DOM 排版失去了坐标控制权，用 Canvas 失去了排版能力。Dagre + 手动定位试图弥合这个裂缝，但只能做到"不重叠"，做不到"好看"。

## 如果重来，可能的路径

1. **不用代码生成，用 draw.io/Excalidraw 手绘模板**。承认这个需求不适合自动化，模板就是手动摆好的布局，用户只填文字。这条路放弃了灵活性换质量。
2. **换约束求解器（如 Cassowary/kiwi.js）**。声明节点间的相对约束（"A 在 B 上方 40px""C 和 D 水平对齐"），让求解器算出满足所有约束的坐标。比 Dagre 的力导向/层级布局更接近排版思维。
3. **Hybrid：手绘骨架 + 数据绑定**。核心布局人工画好（svg 坐标写死），JSON 只驱动文字替换和条件显示/隐藏。相当于模板，但骨架是精确设计的。

以上三条路径工作量都远超当前尝试。项目暂停。
